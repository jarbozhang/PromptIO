一个AMD的工程师，在GitHub上贴出了23万次工具调用的详细数据，指控Claude Code在2月更新后"不可用"。这个issue在Hacker News上拿了1142个赞，627条评论，大量开发者跟帖说"我也是"。

这不是一个普通的吐槽帖。这是一次用数据说话的公开处刑。

## 到底发生了什么

这位工程师的ID是来自AMD的开发者，他的工作涉及复杂的底层工程任务。他长期使用Claude Code作为日常编码工具，积累了海量的使用数据。在2月份Anthropic推送更新后，他明显感觉到输出质量断崖式下降。

但他没有直接骂街，而是做了一件工程师该做的事，量化。

他整理了23万次工具调用的数据，对比更新前后的完成质量、代码正确率、任务成功率。数据清楚地显示，复杂工程任务的完成质量在更新后显著退化。他把这些数据整理成GitHub Issue提交给了Anthropic官方仓库。

Issue标题很直白，"Claude Code is unusable for complex engineering tasks with Feb updates"。

## 社区的反应出乎意料

这个issue在HN上炸了。不是因为一个人在抱怨，而是因为评论区变成了一场大规模的集体验证。

600多条评论里，大量专业开发者报告了相同的体验。不是"感觉变差了"这种模糊表述，而是具体到"之前能一次搞定的重构任务，现在要反复三四轮才能得到可用的结果"。有人说Claude Code开始频繁忽略上下文，有人说它在复杂代码库里的导航能力明显退步。

社区对退化原因的猜测集中在几个方向。一部分人认为是底层模型本身发生了变化，可能是某次静默更新调整了模型参数。另一部分人怀疑是系统提示词被修改了，导致模型的行为模式改变。还有人提出更尖锐的猜测，Anthropic可能在做成本优化，用更便宜的推理配置替代了之前的方案。

VentureBeat也跟进了报道，标题里提到了一个竞品"Goose does the same thing for free"。这句话刺不刺激？你每个月付着订阅费，质量下降了，旁边还有人告诉你免费的替代品干得一样好。

## 中文社区还没意识到的事

我翻了一圈中文技术社区，几乎没有人在讨论这件事。这很奇怪，因为Claude Code在国内开发者中的使用量并不小。

英文社区正在发生的讨论已经超越了"某个工具变差了"这个层面。核心议题是，当你把编码工作流深度绑定到一个AI工具上，你实际上把生产力交给了一个你完全无法控制的黑盒。模型可以在任何时候静默更新，系统提示词可以在任何时候被修改，而你作为用户，连changelog都看不到。

这个issue下面有一条评论让我印象很深，大意是"我们付钱买的不是一个版本的模型，我们付钱买的是一个承诺的能力水平，而这个承诺随时可以被单方面违背"。

## 我的判断

我认为Anthropic大概率做了成本优化相关的调整，同时可能叠加了模型微调的副作用。这不是阴谋论，这是商业逻辑。任何一家烧钱的AI公司都面临推理成本的压力，在用户无感知的前提下降低推理开销，是一个极其诱人的选项。

但这里有一个得罪人的判断，Anthropic的沉默比退化本身更严重。

截至目前，这个1100多赞的issue下面，官方的实质性回应几乎为零。对于一家以"AI安全"和"负责任的AI"为品牌核心的公司来说，在自家产品的核心能力出现争议时选择沉默，这比技术退化本身更让人不安。

如果你对用户的反馈不透明，你凭什么让人相信你对AI安全的承诺是真的？

## 你该怎么办

如果你正在用Claude Code做关键项目，现在就应该做两件事。

第一，建立你自己的质量基线。不要凭感觉判断工具好不好用，记录你的任务完成率、需要手动修改的比例、多轮对话的频率。这位AMD工程师做对了的事情，就是他有数据。

第二，永远不要单押一个AI编码工具。保持对Cursor、Copilot、甚至开源方案如Goose的基本熟悉度。切换成本没有你想象的高，但被锁死在一个退化的工具里的代价，比你想象的大。

AI编码工具的黄金时代可能才刚开始，但"用户无法控制质量"这个问题如果不解决，这个市场的信任基础会比大家预期的更脆弱。
