# Claude Code"变笨了"？393个竖起的大拇指背后，是一场用户集体信任危机

776条评论，393个点赞，72条深度讨论。这不是什么产品发布帖，这是Claude Code用户在GitHub上发起的一次集体控诉。

标题直白得让人心寒："Claude Code在二月更新后，对复杂工程任务已经不可用了。"

## 一个AMD工程师掀了桌子

发帖人Stella Laurenzo，AMD的工程师，干的是LLVM/MLIR和GPU驱动这种硬核系统编程。她不是来吐槽的，她是带着数据来的。

她让Claude自己分析了6,852个会话文件、17,871个思考块、234,760次工具调用，得出了一份让Anthropic没法装看不见的报告。

核心发现只有一句话：Claude从"先读代码再改代码"变成了"直接上手就改"。

具体有多夸张？一月份，Claude每做一次代码修改之前会读6.6次文件。到了三月，这个数字变成了2.0。读代码的比例从46.5%掉到31%，直接写文件的比例翻了一倍。

这不是"偶尔犯傻"，这是工作方式的根本性退化。

## 到底发生了什么

Stella的数据指向了两个时间节点。

第一个：二月中旬，Claude的"思考深度"开始下降。思考内容的中位数长度从约2200个字符暴跌到720个——砍了67%。模型还是同一个模型，但它"想"得少了。

第二个：三月初，Anthropic上线了一个叫"thinking redaction"的功能，把思考过程从UI上隐藏了。三月8号，被隐藏的思考内容占比超过58%。三月12号以后，100%的思考内容对用户不可见。

巧合的是——或者说一点也不巧——用户独立报告质量骤降的日期，恰好是三月8号。

更狠的一组数据：Stella团队做了一个"stop-phrase-guard"脚本，专门抓Claude偷懒、推卸责任、提前终止任务的行为。三月8号之前，这个脚本触发了0次。之后的17天里，触发了173次。

从零到每天10次。

## Anthropic怎么说的

Anthropic的工程师bcherny在帖子里做了正式回应，核心观点是两个：

第一，thinking redaction只是UI层面的变化，"不影响思考本身，也不影响思考预算"。

第二，Opus 4.6引入了"自适应思考"（adaptive thinking），模型会自己决定思考多久，同时默认effort level从high调到了medium（85分）。

他建议用户试试把effort调到high或max，或者设置更短的上下文窗口（`CLAUDE_CODE_AUTO_COMPACT_WINDOW=400000`），又或者用`CLAUDE_CODE_SIMPLE=1`关掉所有自定义配置来排查。

坦率讲，这个回应让社区更愤怒了。

## 用户的真实反馈比官方回复更有价值

我仔细读完了72条评论，把社区里的真实声音归了几类。

**"先改后读"的切身感受。** 多个用户独立验证了Stella的发现。一个用户说："我一直在改CLAUDE.md来对抗这个问题，当时还没意识到原因。"另一个用户说Claude开始频繁用Write整文件覆盖代替Edit精准修改。

**莫名其妙的"时间焦虑"。** 好几个人提到Claude会说"这个任务要花好几天/几周，还是选快速方案吧"。一个产品经理说："我差点就被它说服走了捷径，幸好我逐行检查了。"

**effort调到max也没用。** 多个用户确认即使把effort设到最高，问题依然存在。"reasoning loops"（模型反复推理但不推进）在max effort下依然出现。

**已经在流失用户了。** 这是最让我震惊的：至少有5个活跃评论者表示已经迁移或正在迁移到OpenAI Codex。有人说"I wish I had done it sooner"（我后悔没早点换）。一个用户说同样的任务，Claude Code花两周没做好的，GPT 5.4零样本就搞定了。

当然也有不同声音。一位独立开发者说他用Claude Code独自构建了一个70万行代码的AI网站生成器，觉得完全没问题。他的诀窍是"永远不要期望它自己找到需要的文件，永远自己提供"。

这恰好说明了问题：能用好的人，已经在无意识地帮模型补偿它丢失的"先研究再动手"能力。

## 我的判断

我认为Anthropic犯了一个经典的产品错误：**用大多数用户的平均体验来优化产品，牺牲了最有价值的用户群体。**

bcherny说effort=85是"对大多数用户在智能-延迟-成本曲线上的最佳平衡点"。这话没错。但问题是，愿意为Claude Code Max付月费的用户，恰恰不是"大多数用户"。他们是那些跑30分钟自主任务、写5000字CLAUDE.md、一个周末合并19万行代码的人。

把这群人的体验降级来节省推理成本，是捡了芝麻丢了西瓜。

更让我不安的是Anthropic的回应姿态。当用户拿出234,760次工具调用的数据分析时，官方回复是"你试试调几个环境变量"。当用户问"你们能确认内部没有导致这个问题的变更吗"，回复是一个干巴巴的"Confirmed"。

然后这个issue被关闭了。

Stella在最后一条评论里非常克制地说："它炸得比我预想的厉害得多。"她删掉了一些被断章取义的表述，但坚持了数据本身的结论。

## 如果你也觉得Claude Code"变笨了"

几个立刻能试的事：

- 在settings里把effort level设成high或max（虽然效果有限，但有总比没有好）
- 在CLAUDE.md里强制要求"修改代码前必须先读取目标文件和相关文件"
- 主动控制上下文长度，不要依赖1M窗口自动管理
- 复杂任务拆小，每个子任务开新会话

但说实话，这些都是在帮模型做它以前自己就会做的事。

真正的问题是：当你为一个工具付费，而这个工具在不通知你的情况下降级了核心能力，你的信任要怎么修复？

这个GitHub issue已经关闭了。但它提出的问题，还远没有答案。
