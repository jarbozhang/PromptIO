# 你的AI Agent为什么越用越蠢？因为它的记忆是个无底洞

我最近在做一个 Agent 项目时遇到一个诡异的问题：Agent 的回答质量随着使用时间变长反而在下降。

debug 了两天才意识到——不是模型变蠢了，是上下文被塞满了垃圾。三个月前的 API 调用方式、早就废弃的配置文件路径、甚至已经删掉的数据库表名，全在 context window 里占着位置。Agent 带着一脑子过期信息在做决策。

这不就是数字版的"囤积症"吗？

## 人脑早就解决了这个问题

你的大脑每天处理海量信息，但不会因此越来越卡。原因很简单：海马体会帮你遗忘。

不是被动地丢失信息，而是主动地筛选——重要的记忆被反复巩固，写入长期记忆；没用的东西自然衰减，腾出空间给新信息。你昨天午饭吃了什么大概已经忘了，但第一次线上事故的处理流程你记得清清楚楚。

这就是为什么 GitHub 上一个叫 Hippo 的项目引起了我的注意。它直接从海马体的工作机制出发，给 AI Agent 设计了一套"会遗忘的记忆系统"。

不是又一个 RAG 方案。不是又一个向量数据库封装。

是让 Agent 的记忆像人脑一样，有生有灭。

## 三层架构，对应人脑三种记忆

Hippo 的设计干净得让我意外。三层存储，各管各的：

**Buffer（工作记忆）**——当前会话的临时便签，最多 20 条，会话结束就清空。相当于你工作时在桌上摆的便利贴。

**Episodic Store（情景记忆）**——带时间戳的具体经历，默认 7 天半衰期。你上周踩的一个坑，如果后面没再碰到，它会慢慢淡出。但如果你再次检索到它，半衰期自动延长 2 天。

**Semantic Store（语义记忆）**——从反复出现的情景中提炼出的通用模式。当三条以上相似的情景记忆被触发，系统自动"压缩"成一条语义记忆。就像你不记得每次骑自行车的细节，但"骑车"这个技能已经固化了。

我认为这个三层设计是 Hippo 最值钱的 idea。市面上大多数 Agent 记忆方案要么全存（context 爆炸），要么全靠检索（没有遗忘和巩固机制），Hippo 是我见过的第一个认真对待"遗忘"这件事的工具。

## 几个让我眼前一亮的细节

**错误记忆自动加倍半衰期。** 线上事故和踩坑记录的半衰期是普通记忆的 2 倍。这个设计太懂了——吃一堑长一智是人类最强的学习机制，Agent 也该如此。

**Git commit 驱动的主动失效。** 当你改了代码结构，Hippo 能检测到哪些记忆引用了已废弃的模式，自动削弱它们的权重。我文章开头说的那个"Agent 越用越蠢"的问题，这个功能直接对症下药。

**`hippo sleep` 触发记忆整理。** 名字起得很妙——就像人在睡眠时巩固记忆一样，你可以用 cron 每天跑一次 `hippo sleep`，让系统自动做衰减计算和模式提炼。

**零运行时依赖。** package.json 里 dependencies 是空的，只有一个可选的 `@xenova/transformers` 做嵌入向量检索。SQLite 存储加 Markdown/YAML 镜像，数据可以直接 git track。

这个选择很聪明。Agent 工具链已经够复杂了，再加一堆依赖是给自己找麻烦。

## 社区里的几种声音

项目 3 月中旬上线，目前 256 星，还在早期。从 PR 记录能看出开发节奏很快——工作记忆层、会话连续性、recall 体验优化，一个接一个在迭代。

我注意到社区里有两种态度。

一种认为这是 Agent 记忆的正确方向：现有方案（CLAUDE.md、Cursor Rules、各种 memory bank）本质上都是"全量存储 + 手动维护"，随着项目变大必然崩溃。Hippo 的衰减机制是解决信息过载的根本思路。

另一种持观望态度：衰减参数怎么调？7 天半衰期对快速迭代的项目合适，对维护老系统的场景可能太激进。语义压缩的质量靠什么保证？如果把关键细节"压缩"掉了怎么办？

这两种担忧都合理。

## 我的判断

我认为 Hippo 抓住了一个被严重低估的问题：**Agent 的记忆管理不是检索问题，是信息生命周期问题。**

现在大家疯狂卷 RAG、卷向量检索、卷上下文窗口长度，但很少有人认真想过——不是所有信息都值得被记住。一个永远不会遗忘的系统，本质上就是一个越来越慢的系统。

但我也得说实话：Hippo 目前还是 0.10 版本，星数 256，离生产可用有距离。衰减参数的合理默认值、语义压缩的可靠性、跨项目记忆的隔离和迁移——这些都还需要社区验证。

我更看重的是它背后的设计哲学。不管 Hippo 本身能走多远，"给 Agent 装一个会遗忘的大脑"这个方向一定会成为标配。现在用 CLAUDE.md 手动维护上下文的方式，就像用 txt 文件管理数据库——能用，但终究要被替代。

## 你可以现在做什么

如果你在做 Agent 项目，不一定要立刻上 Hippo，但值得做一件事：审视你的 Agent 现在的"记忆"方案。

Context window 里有多少信息是三个月前写的？有多少引用了已经不存在的代码？你的 Agent 是在用最新的认知做决策，还是在被过期信息拖后腿？

这些问题的答案，可能比你以为的更触目惊心。
