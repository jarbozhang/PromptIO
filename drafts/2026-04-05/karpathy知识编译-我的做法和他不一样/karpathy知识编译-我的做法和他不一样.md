# 我试了Karpathy的知识编译方案，发现他漏掉了最关键的一步

你的收藏夹里有多少"稍后阅读"？我数了一下，光 Obsidian 里就躺了 400 多篇。从来没翻过第二遍。

收藏就是吃灰。这不是自嘲，这是我用了三年笔记工具之后的结论。

然后 Karpathy 上周发了一个 gist，叫 LLM Wiki，核心观点只有一句话：别在查询时检索，要在写入时编译。我照着搭了一周，结果发现他说对了一半——但另一半，他自己可能也没想清楚。

## Karpathy 到底在说什么

先用 30 秒讲清楚他的方案。

传统知识库的逻辑是：存资料 → 需要时检索 → 拼答案。每次都从零开始，上次的思考结果消失在聊天记录里。

Karpathy 的逻辑反过来：每次加入新资料时，LLM 不是存起来等你问，而是**立刻读它、消化它、把信息编译进一个持续维护的 wiki**。更新相关页面，标注矛盾，建立交叉引用。

他的原话：Obsidian 是 IDE，LLM 是程序员，wiki 是代码库。

整套系统三层：底层是只读的原始资料，中间是 LLM 维护的 wiki，顶层是一份 schema 配置文件（告诉 LLM 怎么编译）。日常操作就三件事：摄入新资料、向 wiki 提问、定期让 LLM 做健康检查。

没有向量数据库。没有 Embedding。没有 RAG 管道。

这个思路是对的。但我实际用下来，踩了三个坑。

## 第一个分歧：我不用 Obsidian

Karpathy 用 Obsidian 当前端。我试了两天就放弃了。

原因很简单：Obsidian 是给人看的，不是给 LLM 看的。我 90% 的时间是在和 Claude 对话，不是在 Obsidian 里翻页面。当你的知识库主要消费者是 LLM 而不是你自己的眼睛时，Obsidian 的图谱视图、主题插件、快捷键体系全都是多余的。

我的做法是直接用项目目录 + Claude Code。wiki 就是一堆 Markdown 文件，Claude 启动时读索引文件了解全貌，需要细节再读具体文件。不需要任何额外工具。

你可能会说，那图谱可视化呢？说实话，图谱好看但没用。你上次因为看了图谱视图而产生了新想法，是什么时候？

## 第二个分歧：只对话，不收藏

Karpathy 的方案有一个 `raw/` 目录，用来存原始资料。他的流程是先收集，再编译。

我把这步砍了。

看到一篇有价值的文章，我不收藏链接，不下载 PDF，而是直接把内容扔给 Claude 对话。聊完之后，有用的结论直接落到知识库里。原始链接？不留。

为什么？因为收藏这个动作本身就是一个陷阱。你以为"存起来以后看"，但"以后"永远不会来。Karpathy 的 `raw/` 目录本质上还是一个收藏夹，只不过多了一步自动编译。但如果你从一开始就不收藏，而是当场消化，这个问题就不存在了。

我的知识库里没有任何"待处理"的东西。每一条都是已经被思考过的结论。

## 第三个分歧：以项目为单位，不搞统一大库

Karpathy 演示的是一个统一的个人知识库。我试过，到第三周就崩了。

问题出在哪？一个统一的大库意味着 LLM 每次编译新内容时，都要理解整个库的上下文。你往里面同时塞了 RAG 技术文档、产品需求笔记、读书摘要，LLM 建立的交叉引用就会变得越来越离谱——它会试图把一本小说的读后感和你的 API 设计文档关联起来。

我的做法：**每个项目一个独立知识库**。做内容管线项目时，那个项目目录下就有一个 `kb/`。研究 Agent 架构时，另一个目录另一个 `kb/`。

好处是每个知识库的范围有界，LLM 不会乱建连接，维护成本可控。坏处是跨项目的知识不会自动关联。但说实话，你真的需要你的烘焙笔记和 Kubernetes 配置互相引用吗？

## Karpathy 说对了什么

虽然我改了三个地方，但他的核心洞见是真的牛。

**"编译"这个比喻精准到可怕。** 传统笔记是原始文本的堆砌，查找靠全文搜索。RAG 是在原始文本上加了一层索引，查找靠向量匹配。而编译是把原始文本**转化**成了新的、结构化的产物。每次编译都在积累，知识在复利增长。

**自动 Linting 是杀手级特性。** 定期让 LLM 给知识库做体检：找矛盾、找孤儿页面、找信息空白。相当于你的知识库有了免疫系统。我现在每周跑一次，每次都能发现之前没注意到的认知盲区。

**"临时 Wiki"模式极具想象力。** 针对一个具体问题，LLM 临时构建一个小型知识库，跑几轮自检，输出一份完整报告。用完即弃，或者把精华沉淀回主库。这已经不是笔记管理了，这是自动化研究。

## 英文社区讨论里藏着的好东西

HN 评论区有几个方向国内基本没人聊：

**跨文档矛盾检测。** 两篇论文对同一个问题结论相反？LLM 在 Linting 时自动标出来。做文献综述的人听到这个应该会兴奋。

**结构化实体提取。** 不只是写摘要，而是自动抽取人物、机构、方向，构建关系图。丢进去 50 篇 AI 论文，自动生成"谁在跟谁合作"的关系网络。

还有一条评论让我印象很深：对于没有机构图书馆访问权限的独立研究者，这套工具等于一个 24 小时在线的研究助理。

## 我的判断

Karpathy 把范式说清楚了——知识库应该像代码仓库一样管理，有输入、有编译、有产物、有测试。

但具体怎么做，你不应该照搬任何人的方案。

我认为最关键的一步，Karpathy 没有展开讲：**你得先有明确的工作流和需求，再搭知识库**。否则你只是用一个更高级的方式在"收藏吃灰"。先想清楚你每天在重复哪些思考、哪些查找，再决定知识库的结构。

国内那些"AI 笔记"产品，90% 还停留在"帮你总结一下"。而 Karpathy 展示的是让 AI 帮你**运营**一个会自己生长和纠错的知识库。这中间差了一整个范式。

你现在就可以试：拿你手头一个具体项目，建一个 `kb/` 目录，写一份 schema 告诉 LLM 怎么编译，然后把项目相关的资料喂进去。不用搞什么统一大库，从一个小项目开始。

你的笔记工具是在帮你积累，还是在帮你囤积？评论区聊聊。
