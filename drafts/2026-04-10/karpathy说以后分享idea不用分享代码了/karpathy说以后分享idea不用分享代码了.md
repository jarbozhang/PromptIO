Karpathy 上周发了一条推文，拿了 52000 个赞。内容不是什么新模型发布，不是什么论文突破，而是他最近在用 LLM 给自己搭知识库。

两天后他又发了一条，25000 赞，说了一句让我愣了好几秒的话，"以后分享东西不用分享代码了，分享 idea 就行，让对方的 Agent 自己去造。"

然后他真的这么做了，把自己的 LLM Wiki 方案写成了一个 GitHub Gist，叫 "idea file"。不是一个 repo，不是一个 npm 包，不是一个可以 `git clone` 跑起来的项目。就是一篇描述性的文档，讲清楚这个东西的架构思路、核心操作、设计理念。

你拿到这个 idea file，丢给你的 Claude Code 或者 Cursor，它就能根据你的具体需求把整个系统搭出来。

这事有意思在哪？

## 先说他在造什么

Karpathy 最近的 token 消耗，越来越少花在写代码上，越来越多花在"操作知识"上。他的工作流是这样的，把各种文章、论文、数据集丢进一个 `raw/` 目录，然后让 LLM 把这些原始资料"编译"成一个 wiki，就是一堆 markdown 文件，有摘要、有交叉引用、有概念页面、有对比分析。

用 Obsidian 当前端看这些文件。

他自己的一个研究 wiki 已经积累了大约 100 篇文章、40 万字。到了这个规模，直接问 LLM 复杂问题，它能自己去 wiki 里翻资料、组织答案。他说本来以为需要搞 RAG，结果发现 LLM 自己维护的索引文件和摘要已经够用了。

坦率讲，这个想法本身不算特别新。很多人在用 Obsidian + LLM 搭类似的东西。但 Karpathy 把它提炼到了一个很精炼的三层架构，Raw Sources（不可变的原始资料），Wiki（LLM 生成和维护的 markdown），Schema（一个配置文件，告诉 LLM 怎么维护这个 wiki）。

关键在第三层。Schema 这个东西其实就是 CLAUDE.md 或者 AGENTS.md，它把 LLM 从"通用聊天机器人"变成"有纪律的知识库管理员"。

## idea file 才是真正的炸弹

但我觉得更值得关注的不是 LLM Wiki 本身，而是他选择的分享方式。

他没有开源一个项目。他发了一个 Gist。

这个 Gist 故意写得比较抽象，不指定具体的目录结构、不规定用什么工具、不写死实现细节。因为他的逻辑是，你的需求和我不一样，你的 Agent 会根据你的情况定制出不同的实现。

你想想看，这其实是在重新定义"分享"这件事。

过去十年开源社区的范式是什么？我写了个好用的东西，打包成一个 repo，你 clone 下来，按 README 装依赖、跑起来，然后根据自己的需求改。改的过程通常很痛苦，因为别人的代码里满是你不需要的假设和设计决策。

Karpathy 提出的范式是，我不给你我的实现，我给你我的想法。你的 Agent 从零开始，按照你的需求构建。没有别人的技术债，没有不适合你的架构决策，没有你不需要的功能。

说真的，这个想法一开始让我有点不舒服。

## 这事经得起推敲吗

我的第一反应是，这不就是写文档吗？一个 idea file 和一篇好的技术博客有什么区别？

想了想，区别确实有。

传统技术博客的读者是人。idea file 的读者是 Agent。所以它的写法、结构、详细程度都不一样。它不需要解释"为什么"，不需要循序渐进的教学，它需要的是足够清晰的架构描述和约束条件，让 Agent 能准确地把意图变成代码。

Karpathy 自己也说了，这个 Gist 的讨论区可以让别人调整 idea 或者贡献自己的想法。这就有意思了，idea file 不是一锤子买卖，它可以像 RFC 一样演进。

社区里有人已经开始跟进了。一个叫 Farza 的开发者做了 "Farzapedia"，就是按照 Karpathy 的 idea 让 Agent 搭了一个个人 wiki。Karpathy 看了很喜欢，专门发推说这是一个很好的实践案例，还顺便阐述了他对"个性化 AI"的理念，数据是你的、格式是通用的、AI 是可替换的。

但我也在想反面。

一个 idea file 能承载的信息密度，远不如一个经过打磨的代码库。真实系统里那些边界情况、性能优化、错误处理，idea 里写不出来。Agent 现在的水平，处理一个两百行的小项目没问题，但给它一个模糊的 idea 让它从头搭一个生产级系统？

我认为还差得远。

## 但方向是对的

可能有些想法还不成熟，但我的判断是，idea file 这个概念指向了一个真实的趋势，软件的分发单位正在从"代码"向"意图"迁移。

现在已经有苗头了。Claude Code 的 CLAUDE.md、Cursor 的 .cursorrules，其实都是 idea file 的雏形，用自然语言描述你想要什么，让 Agent 去执行。只不过现在这些文件的粒度还很小，是项目级别的。Karpathy 把它推到了"产品级别"，一个完整的软件概念可以浓缩成一个 idea file 来传播。

我自己也还在摸索。最近在用 Claude Code 搭类似的知识管线，体验是，LLM 确实很擅长做那些繁琐的知识整理工作，分类、交叉引用、写摘要、维护索引。这些事以前我都知道该做，但就是懒得做。现在 LLM 给你做了，知识终于能"复利"了。

但要从 idea file 直接生成一个完整的、符合你需求的系统？目前还是得人在旁边盯着。Agent 会漏东西，会做出你不想要的设计决策，会在你没注意的地方引入问题。

所以现阶段，idea file 更像是一个超级详细的需求文档，不是一个可以无脑执行的蓝图。

但这已经够改变游戏规则了。想想看，一个高手花一下午写的 idea file，可能比他花一个月写的开源项目对你更有用。因为开源项目里 80% 的代码是为他的场景设计的，而 idea file 里 100% 的思考都可以被你的 Agent 适配。

这可能是我今年看到的最有启发的一条推文。不是因为 LLM Wiki 这个产品有多厉害，而是 "idea file" 这三个字，可能会成为 Agent 时代知识传播的新单位。

你愿意试试把自己最得意的项目浓缩成一个 idea file 吗？
