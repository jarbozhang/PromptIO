# 偷师Claude Code缔造者：顶级AI编程Agent的工作流拆解与复现

你以为AI编程就是对着Cursor或者Copilot打几句“帮我写个Todo List”？最近，Claude Code的灵魂人物Boris Valkov亲自公开了自己的日常工作流，整个开发者社区直接炸了锅。

看完他的演示，我才意识到大多数人对AI编程的用法完全是错的——包括我自己。如果你正在做AI应用或者重度依赖AI写代码，他这套“不讲武德”的Prompt和工作流，会直接颠覆你接下来的开发效率。

## 核心心法：你才是架构师，AI是实习生团队

Valkov的工作流里最反直觉的一点：他从来不让AI从头到尾一口气写完一个功能。

绝大多数人的用法是丢一个大需求，然后苦等两分钟，看着AI生成几百行代码，最后发现跑不通又回来骂模型太笨。Valkov的做法完全相反，他把自己定位为“严格的技术架构师”，把AI当成一个极其聪明但容易跑偏的实习生。

想象你正在做一个RAG应用，突然发现检索模块的准确率很低。Valkov的思路绝对不是直接让AI“重构检索模块”。他会先让AI去读现有的代码结构，然后一步步给出非常具体的指令。

这背后的核心逻辑是：**上下文的精准控制**。AI不需要知道你的整个业务大盘，它只需要知道当前这10分钟内它要解决的具体问题。

## 实操拆解：Valkov的“剥洋葱”Prompt法

Valkov最常用的技巧，我把它总结为“多轮剥洋葱法”。这不是简单的聊天，而是一套经过精心设计的试探与推进流程。

**第一步：强行让AI阅读代码，闭上嘴巴。**
不要让AI上来就动手改。先用Prompt强行限制它的行为。
```bash
# 关键限制词：不要生成代码
> 请阅读 src/rag/retriever.py 这个文件。
> 分析当前使用的向量检索逻辑，列出3个可能导致准确率下降的瓶颈。
> 在我给出下一步指令前，绝对不要生成任何修改代码。
```

**第二步：隔离讨论方案，杜绝干扰。**
等AI分析完，再让它基于分析给方案。这时候依然不写代码。
```bash
> 针对你提出的第2个瓶颈（Chunk切分策略不合理），
> 给出3种可选的替换方案，对比它们的计算开销和实现复杂度。
```

**第三步：指定方案，限定范围，开始编码。**
只有到了这一步，才真正让AI动手。并且明确告知它改动的影响范围。
```bash
> 采用方案A（递归字符切分）。
> 你只能修改 retriever.py 文件中的 chunk_data 函数。
> 保持原有的函数签名和返回类型不变，直接给出替换代码。
```

短指令，强限制，单点突破。这就是顶级工程师驾驭AI的秘诀。

## 踩坑与真实体验：别被“自然语言编程”的噱头骗了

我自己在复现这套工作流时，最大的感受是：**极度消耗脑力**。

别被“自然语言编程”的噱头骗了，你写Prompt的思考密度，比直接写代码还要高。你必须时刻清楚项目的整体架构，精准评估每一步改动的副作用。如果你自己脑子里没有架构图，这套工作流根本玩不转。

Reddit和Hacker News上的讨论非常有意思。很多开发者抱怨说：“我试了Valkov的方法，确实生成的代码质量高了，但我一天下来的产出反而降低了。”

为什么会这样？因为他们把时间都花在了写Prompt和审查上下文上。

这就引出了一个核心的分水岭：**这套工作流只适合复杂的重构、架构设计和核心逻辑编写。**

如果你只是写个增删改查的API接口，或者写个简单的脚本，用这套流程纯属给自己找麻烦。对于那些明确、简单、机械的任务，直接让AI一口气写完，然后人工Review一遍，效率反而更高。

我自己尝试后的真实体感：
*   **简单任务（占比70%）**：直接给需求，一口出代码，人工Review。（耗时：快）
*   **复杂架构（占比20%）**：套用Valkov的剥洋葱法，严格控制上下文。（耗时：中等，但极少返工）
*   **玄学Debug（占比10%）**：把报错日志和相关代码喂给AI，让它扮演“无条件配合的助理”来共同排查。（耗时：看运气）

情绪上的转折点发生在第三天。前两天我觉得这方法太繁琐，直到周五下午我需要重构一个复杂的支付状态机。如果按以前的习惯让AI直接写，大概率会在测试环境里跑出各种边界Bug。但我用了“剥洋葱法”，一步一步隔离上下文，半小时搞定，一次通过，没有任何返工。

那一刻我懂了Valkov的良苦用心。

## 信息差洞察：英文社区正在讨论什么？

这个信息目前只在英文社区的深度讨论串里流传，国内几乎没人聊过：

**核心差异不在于模型，而在于“工作记忆”的管理。**

Hacker News上一位前Google工程师指出，Valkov这套流程的本质，其实是把人的“工作记忆”剥离出来，用外部的文档和精准的Prompt代替。人在审查代码时，大脑的短期记忆是极其有限的。当你看着AI生成的300行代码时，你其实只能注意到其中最显眼的几行。

Valkov的做法是：用短上下文换取高注意力。每次只让AI生成一小块，你集中全部注意力去审查这一小块。审查通过，再推进下一块。

这其实是传统的“微操作”在AI时代的复苏。很多国内开发者还在追求“一键生成全栈App”的幻想，但英文社区的一线工程师们已经回归现实：把AI当做需要精细控制的打字机，而不是无所不能的架构师。

国内很多团队还在迷信所谓的“全自动化Agent”，砸大价钱搞复杂的编排逻辑。而顶级黑客在做的，是用最简单的工具加上最严格的纪律，实现单点效率的极致提升。这种认知差，才是最致命的。

## 我的判断

我认为，Claude Code创始人的这套工作流，**值得每一个正经写代码的人立刻开始尝试，但别盲目照搬。**

这玩意不是银弹。如果你是一个连基本设计模式都不清楚的初级程序员，这套方法救不了你，因为AI给出的多个方案你根本无法评估优劣。你需要先积累足够的工程常识。

但对于有一定经验的工程师，尤其是那些正在做AI Native应用、每天需要处理复杂业务逻辑的人，这套“剥洋葱法”是当下的最优解。

谁应该立刻学？正在做复杂系统重构、后端架构设计、或者需要经常处理遗留代码（屎山）的人。

谁可以忽略？如果你主要的工作是做前端UI还原、写简单脚本，或者只是偶尔用AI查查语法，继续用你的简单Prompt就好，别给自己加戏。

结合国内的AI开发生态，很多团队在搞“Agentic Workflow”，画各种复杂的DAG图。我的建议是：停下来看看Valkov的做法。与其花一个月时间调通一个复杂的自动化Agent链路，不如先把“人机协同”的Prompt基本功练好。工具再智能，最终决定代码质量的，还是操控工具的那个人的工程素养。

## 行动指南：30分钟跑通这套心法

别光看文章，今晚就动手试一次。找一个你正在做的项目里稍微复杂的函数，按照下面的步骤走一遍：

1.  **选择目标**：挑一个超过50行、逻辑稍微有点绕的函数。
2.  **强制阅读**：把函数代码贴给AI，要求它“只分析，不生成代码，列出潜在问题”。
3.  **隔离讨论**：针对其中一个问题，让它给出2-3种修改方案并对比。
4.  **精准生成**：选定方案，明确告诉它只能改动哪几行，保持其他部分不变。

在这个人机协同的磨合期，你最想解决的痛点是什么？是让AI理解你的屎山代码，还是让它严格按照你的架构规范输出？欢迎在评论区聊聊你的真实体验，如果你身边有正在被AI编程折磨的同事，把这篇文章转发给他。
