# 我让AI通宵写代码的秘密：测完这三个神级工具，我终于搞懂了让Agent跑满整晚的底层逻辑

你可能遇到过这种情况：让 Claude Code 写个稍微复杂点的功能，开始几分钟看着还行，跑去泡杯咖啡回来，发现它已经陷入了死循环，或者悄悄引入了一堆肉眼看不见的 Bug。我以前也是这样，直到我深度研究了国外开发者圈子里爆火的三套工具。

我花了一个周末，逐行对比了 Garry Tan（YC 掌门人）的 **gstack**、Jesse Vincent 的 **Superpowers**，以及 Every Inc. 的 **Compound Engineering (CE)**。这三兄弟在 GitHub 上狂揽了几十万 Star，外媒铺天盖地报道。但我发现，国内基本没人真正说清楚它们到底怎么用，绝大多数人只是装了一个就以为自己"AI 全自动编程"了。

说实话，这不是三个竞争对手，而是解决"AI 长时间自主编程"的三个不同层。错过了哪一层，你的 Agent 依然是半残废。

### 为什么你的 AI 编程只能跑 5 分钟？

先用一个餐厅的比喻来解释为什么大多数人的 AI 编程体验极差。

想象你开了一家餐厅。你雇佣了一个天才厨师（大模型）。第一种失败：你什么都不问，直接让他做菜，结果做出来一桌子没人点的菜（没有**决策**层）。第二种失败：他做完了自己尝了一口说"绝了"，但其实咸得发苦（没有**质检**层）。

第三种失败最致命：今晚厨师踩了灶台漏气的坑，折腾了三小时才修好。第二天换了个新厨师来，又在同一个灶台上摸黑折腾三小时。因为你没有**交接记录和沉淀**。

这三个痛点，恰恰对应了去年 11 月 Anthropic 官方工程博客里提出的长周期 Agent 架构核心：构建者评估自己的工作，通常会盲目乐观。做菜的人和尝菜的人，必须分开。

### gstack：最强主厨 + 独立试吃员

gstack 解决的是"到底该不该做"和"做没做对"的问题。它是 YC 掌门人 Garry Tan 的杰作。他自己号称用这套东西，在全职运营 YC 的同时，60 天内交付了 60 万行生产级代码（当然，这是他自己统计的数字，你的情况可能有所不同）。

它的杀手锏是 `/plan-ceo-review` 和 `/plan-eng-review`。CEO Review 从产品角度灵魂拷问你："这玩意儿值得做吗？" Eng Review 从架构角度警告你："照这么写，上线肯定炸。" 这两道关卡都通过了，代码才会开始动。

更厉害的是 `/qa`。别被名字骗了，它不是看代码有没有语法错误。它会启动一个真实的浏览器，像真实用户一样去点击你的网站。Anthropic 的测试也证明了，这种基于浏览器的端到端测试，效果远超代码层面的静态检查。代码看起来"没问题"和实际用起来没问题，中间差了十万八千里。

在实操中，启动项目前我强烈建议用这个 Prompt：**"Interview me until you have 95% confidence about what I actually want, not what I think I should want."**

让 AI 来采访你，而不是你给 AI 提需求。绝大多数项目的失败，根本不是代码写错了，而是从一开始就没人搞清楚到底要做什么。

### Superpowers：厨房的流程操作手册

Superpowers 拥有超过 12 万 Star，它的核心是把"随意聊天"变成了"标准化流程"。

它的核心周期是：头脑风暴 → 计划 → 执行 → 审查。它帮很多人完成了从胡乱对话到流程化使用 AI 的飞跃。就像从厨师看心情做菜，变成了厨房里有明确的备菜清单和烹饪步骤。这绝对是巨大的进步。

但它有一个致命缺陷：没有记忆。

每次会话结束，所有的经验、踩过的坑、试错的过程，都随风消逝了。下一次开新的会话，Agent 就像一个失忆的厨师，昨天刚修好的漏气灶台，今天又要重新摸黑检查一遍。

### Compound Engineering (CE)：缺失的复利层

这就是为什么我最终把 CE 叠加到了我的工作流里。如果说前两个工具让你"今晚能做出好菜"，CE 解决的是"这家餐厅如何持续开下去"。

CE 的流程是：头脑风暴 → 计划 → 工作 → 审查 → **复利沉淀**。

前四步和 Superpowers 类似，但深了一个量级。计划阶段，它不会在当前对话里拍脑袋，而是生成并行的子 Agent，去深挖你的项目历史、扫描代码库模式、读 Git 提交记录。审查阶段更恐怖，它会启动一个动态审查委员会，最少 6 个角色（正确性、安全性、性能、可维护性等）同时给你的代码出独立报告。

但真正的王炸是最后一步：`/ce:compound`。

这就是它名字的来源——**复利**。每次修完一个 Bug 或完成一个功能，运行这个命令，它会并行启动 5 个子 Agent：分析上下文、提取解决方案、查重已有文档、制定预防策略、分类打标签。最后全部自动生成结构化文档存入 `docs/solutions/` 目录。

用人话讲：你的 Agent 每天下班前，都会写一份极其详尽的交接总结。下次启动任何新任务时，Agent 都会先把这些总结全部读一遍。比如三周前你花了几小时解决了一个 Edge Runtime 兼容性问题，Compound 当时已经自动记录了。三周后类似问题再次出现，计划阶段的 Agent 会自动检索到："我们之前踩过这个坑，解决方案在这里。" 几小时的调试被压缩成了几分钟。

Anthropic 官方主张的进度文件是线性的，像昨晚留给早班的便签。而 CE 是一本可检索的菜谱大全。一个是线性传递，一个是指数级复利。

### 别被 Github README 骗了：真实踩坑经验

把这三个工具叠加在一起用很爽，但如果你是新手，千万别一上来就全装。

我刚开始就踩了这个坑。多个技能包同时存在，会产生严重的流程冲突和命令覆盖。有时候 Agent 执行到一半突然卡住，分不清到底在走哪套审查流程。

最稳妥的路径是：先选定一个核心框架（我推荐从 gstack 或 CE 中选一个），跑通你的第一个项目。等你理解了它的脾气，再慢慢把其他工具的命令嵌进来。

还有一个隐藏的坑。Claude Opus 4.6 目前有 100 万 Token 的上下文窗口，测试版可以直接用。如果你的项目能一次性塞进去，确实很爽。但别过度依赖上下文窗口。即使上下文再大，Agent 还是会遗漏细节。Anthropic 官方依然强调外部状态文件（如 `claude-progress.txt`）才是主要的协调机制。别偷懒，让 Agent 老老实实把状态写进文件里。

### 中文圈还没人注意到的信息差

最近在 Hacker News 和 Reddit 的讨论串里，英文开发者圈已经形成了一个强烈的共识：**2026 年 AI 编程的核心竞技场，早就不是模型写代码的能力了，而是 Agent 的记忆架构和工具编排能力。**

国内目前对 AI 编程的讨论，大多还停留在"提示词怎么写"和"Cursor 又更新了什么快捷键"的阶段。这其实慢了半拍。当别人已经在研究如何让 Agent 自动积累项目经验、跨会话避免重复踩坑时，我们还在手动复制粘贴上下文给 AI。

这三套工具的组合使用，目前在国内几乎没有人系统性地聊过。很多团队可能知道 Cursor 或者是 Devin，但不知道自己可以通过开源工具，在本地搭建起比这些商业软件更强大、更符合自己团队业务逻辑的长效 Agent 工作流。

### 我的判断与行动建议

我认为，如果你是一个独立开发者或技术合伙人，手里有大量需要开发的需求，**CE 是你今天就必须尝试的工具，没有之一。** 它的知识复利机制，是目前我看到的最接近"拥有一个不断成长的全栈工程师"的方案。

如果你是一个技术Leader，正在头疼如何让团队里初级开发者用 AI 不写出一堆垃圾代码，**gstack 是你的救星。** 它的双关卡决策和真实浏览器 QA，能挡住 80% 的架构错误和功能遗漏。

如果你只是偶尔用 AI 写写脚本，那暂且别折腾了，这些工具的学习成本和编排复杂度对你来说杀鸡用牛刀。

行动起来吧。去 GitHub 把 CE 克隆下来跑个 Demo。试着让它帮你修一个多日未解的 Bug，然后跑一遍 `/ce:compound`。看看它在 `docs/solutions/` 里生成的报告，你就会立刻明白什么叫"知识复利"。

如果觉得这篇文章对你有启发，转发给你团队里正在被 AI 写代码折磨的同事。你也可以在评论区聊聊：你目前在用 AI 编程时，遇到的最大痛点是什么？我们看看能不能一起编排个解决方案。
