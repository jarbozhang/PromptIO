# 别瞎装 Claude Code 插件了：我对比了 gstack、Superpowers 和 Compound，发现它们根本不是竞品

最近逛国外的 AI 开发者社区，你会发现有三个 Claude Code 工具火得一塌糊涂：Garry Tan（YC 掌门人）的 gstack（54.6K ⭐），Jesse Vincent 的 Superpowers（121K ⭐），还有 Every Inc. 的 Compound Engineering（11.5K ⭐）。

国内很多同学看名字以为它们是同类竞品，随便挑一个装上就觉得万事大吉。但我把这三个仓库全部拉下来跑了一遍，发现了一个反直觉的结论：**这根本不是三个竞争工具，而是三个完全不同的层级。**

如果你正在用 Claude Code 做项目，没搞懂它们的区别就乱装，轻则功能重叠，重则流程冲突。想象一下，你雇了一个米其林大厨（gstack），流程手册也写得漂漂亮亮（Superpowers），结果后厨连个“错题本”都没有。第二天新换班的厨师，又把昨天踩过的雷重新踩一遍。

今天这篇文章，我就用人话帮你拆解这三个工具的核心差异，以及怎么把它们叠起来用。

## 衡量标尺：Anthropic 的“套件架构”

为了把这三者放在同一个维度里比较，我借用了 Anthropic 官方工程团队在 2025 年底发布的一篇关于“长时间运行 Agent 套件设计”的博客作为标尺。

我可以把这个架构比喻成一家餐厅的四个核心职责：
1. **主厨定菜单（规划 Planning）**：决定做什么、不做什么。
2. **后厨团队炒菜（执行 Execution）**：把活干出来。
3. **独立的试菜员把关（评估 Evaluation）**：自己做好的菜，自己尝必然觉得香。评估和执行必须分离。
4. **打烊交接班记录（跨会话状态 Cross-session state）**：今晚踩的坑，明早接班的人必须知道。

其中最核心的洞察是：**自己构建代码的 Agent，对自己的工作成果普遍存在过度乐观。** 只有把“造物主”和“质检员”拆开，才能跑出高质量的代码。

搞懂了这个框架，我们再来看这三个工具，你会发现它们各自的“重力中心”完全不同。

## gstack：决策与质检的双闸门

gstack 的核心是帮你解决“做不做”和“做得对不对”的问题。

它最强的地方在于主厨决策和真实 QA。用过 `/plan-ceo-review` 和 `/plan-eng-review` 这两个命令你就知道，它会从“产品角度值不值得做”和“架构角度会不会爆炸”两个维度对你进行灵魂拷问。这两道闸门全通过了，代码才会真正开始写。

在真实测试中，我发现 gstack 的 `/qa` 命令是极大的亮点。它不像普通的代码审查只看“代码层面没报错”，而是会直接打开一个真实的浏览器，像个真实用户一样在你的网站上点来点去。Anthropic 自己的测试也证明了，这种基于真实浏览器的端到端测试，比单纯依赖代码检查的性能要高得多。

这里分享一个我常用的实用技巧。在真正跑项目之前，先用这句提示词：

> “我准备开始这个项目了。采访我，直到你有 95% 的把握知道我**真正**想要什么，而不是我以为我想要什么。”

别急着指挥 AI 干活。让 AI 来疯狂反问你，这是避免项目后期推倒重来的最佳手段。

## Superpowers：没有记忆的流水线手册

Superpowers 拿到 121K Star 绝非偶然。它把“头脑风暴 → 计划 → 执行 → 审查”的流程打磨得极其丝滑。

如果说以前你用 AI 写代码是“全凭感觉瞎聊”，那 Superpowers 就是给后厨引进了一套标准的流程手册和备菜清单。它确实把大批开发者从“瞎聊模式”拉到了“工程化模式”。

但我实际用下来，发现它有一个致命的盲点：**它没有跨会话的记忆沉淀机制。**

你在这个会话里调试好的经验、踩过的坑，全都被锁死在了当前的上下文里。下一个会话一开启，又是一张白纸。这就好比餐厅里大家都按标准流程炒菜，但下班后没人写交接记录。第二天换批人来，该糊锅的菜还是会糊。

## Compound Engineering：你急需的那本“错题本”

这就是 Compound Engineering（CE）切入的断层。

CE 的工作流也是：头脑风暴 → 计划 → 执行 → 审查，但它在两个环节深入到了极致，并且加上了至关重要的“第五步”。

**计划阶段：** CE 不是凭空写计划，而是会唤醒并行的“研究子智能体（Subagents）”，去翻阅你的项目历史、扫描代码库模式、阅读 Git 提交记录。这就像新来的大厨在定明天菜单前，先把过去三个月的“顾客退菜记录”翻个底朝天。

**审查阶段：** 它不是随便找个质检员说一句“代码没报错，还行”。CE 会拉起一个动态审查团队，**最少 6 个独立的审查员**，涵盖正确性、安全性、性能、可维护性、对抗性等维度，各自出独立的报告。这就像同时请了美食评论家、卫生局检查员和大众评审来给同一道菜打分。

**但真正让它具有颠覆性的是第五步：`/ce:compound`。**

这就是它名字里 Compound（复利）的由来。当你修完一个 Bug 或完成一个功能，跑一下这个命令，它会并行唤醒 5 个子智能体：
- **上下文分析器**：提取问题类型和涉及组件
- **解决方案提取器**：记录什么方法行不通，最终解法是什么
- **相关文档查找器**：检索已有知识库，避免重复记录
- **预防策略师**：总结未来怎么预防此类问题
- **类别分类器**：打上标签，方便未来结构化检索

这 5 个智能体跑完后，结果会被合并写入 `docs/solutions/` 目录。

大白话就是：**你的 AI 下班前，强制要求它写一份结构化的工作总结和错题本。下次任何任务开始前，第一件事就是先把这些总结全部读一遍。**

Anthropic 官方提倡的进度文件就像是“后厨值夜班留给早班的便签”，线性且单薄。而 CE 的 `docs/solutions/` 是一本可搜索、持续迭代的“百年老店独家配方簿”。

一个是线性延续，一个是指数级复利。用得越久，你的 Agent 就越懂你的项目。

## 三个层级到底怎么叠？

搞清楚了各家的重心，我们用一张表总结：

| 核心层级         | 对应工具              | 餐厅比喻           |
| :----------- | :---------------- | :------------- |
| **决策（做不做）**  | gstack            | 主厨定菜单          |
| **规划（怎么做）**  | CE `/ce:plan`     | 读取历史客诉后再备料     |
| **执行（写代码）**  | CE `/ce:work`     | 后厨按单炒菜         |
| **审查（做得对吗）** | CE + gstack `/qa` | 多维度打分 + 真实用户试吃 |
| **知识沉淀（避坑）** | CE `/ce:compound` | 世代相传的配方簿       |

注意，它们之间有功能重叠（比如审查环节），但核心重心完全不同。

## 我的实战组合工作流

如果你刚开始用，千万别三个一起上。多个技能包混用极容易产生流程冲突和命令重叠。先挑一个（gstack 或 CE）跑通。

对于有经验的老手，这是我目前测试下来最顺滑的“全家桶”流程：

1. **理清需求**。用前面提到的 95% 置信度提示词，让 AI 反问你。
2. `/office-hours`（gstack）。描述你要做的事，接受它的灵魂拷问。
3. `/plan-ceo-review`（gstack）。过产品关：这玩意儿值得做吗？
4. `/plan-eng-review`（gstack）。过架构关：这么做以后会炸吗？
5. `/ce:brainstorm`（CE）。探索需求细节，沉淀为规格说明书。
6. `/ce:plan`（CE）。研究 Agent 扫描项目历史，出详细实施方案。
7. `/ce:work`（CE）。带任务追踪地疯狂输出代码。
8. `/ce:review`（CE）。拉起 6 个以上的独立审查员进行代码级找茬。
9. `/qa`（gstack）。在预发环境启动真实浏览器，模拟用户真实点击测试。
10. `/ce:compound`（CE）。记录今天的所有经验教训，写入配方簿。
11. **发布上线**。下一次从第 1 步重新开始时，你的第 6 步就已经拥有了今天的记忆。

第 1 到 4 步保证你在“做对的事情”。第 5 到 9 步保证你“把事情做对”。第 10 步保证你下一次能做得更快。

## 我的判断

我认为，目前国内中文社区对 AI 编程的讨论，还停留在“哪个模型跑分高”或者“怎么写好提示词”的层面。

但英文圈子已经在死磕“工程化闭环”和“知识复利”了。这也是为什么 Compound Engineering 这种工具能在硬核开发者圈子里疯狂传播的原因。

如果你的项目只是个一次性跑完就扔的脚本，装个 Superpowers 足够让你爽一把。但如果你在长期维护一个大型项目，或者你在做 ToB 的 AI 落地应用，没有知识沉淀层（Knowledge Accumulation）将会是你团队效率的致命瓶颈。

你的 AI 每天都在写代码、修 Bug、跑测试。干完活之后，这些经验去了哪里？

如果答案是“散落在各个对话历史里，下次接着踩坑”，那你现在就该去 GitHub 拉一下 CE 的代码了。

如果觉得这篇拆解对你有启发，转发给你团队里正在搞 AI 研发的同事。你觉得现在的 AI 编程，最大的瓶颈是在规划、执行，还是经验沉淀？欢迎在评论区聊聊。
