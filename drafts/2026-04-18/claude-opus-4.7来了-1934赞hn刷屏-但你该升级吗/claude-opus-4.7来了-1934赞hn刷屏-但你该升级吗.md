# Claude Opus 4.7来了，1934赞HN刷屏，但你该升级吗

1934个赞，1421条评论。Claude Opus 4.7 发布当天直接冲上 Hacker News 榜首，这个数字已经接近年度级别的讨论热度了。

但翻完评论区，我发现社区的态度远没有点赞数那么统一。

有人说这是"第一个真正理解我在做什么的模型"，也有人说升级后代码质量反而变差了。到底发生了什么？

## Anthropic 给了什么

先快速过一遍官方数据。SWE-bench Verified 从 80.6% 涨到 87.6%，SWE-bench Pro 从 53.3% 跳到 64.3%，编码基准提升了 13%。文档理解能力更夸张，图表识别从 13.5% 直接拉到 55.8%。视觉处理上限扩大到 2576 像素长边，大概 375 万像素，是之前的三倍多。

GDPval-AA 上拿到 1753 Elo，号称对 GPT-5.4 胜率约 60%。

价格没变，还是每百万输入 token 5 美元、输出 25 美元。

数字很漂亮。但数字从来不是全部。

## 真正值得关注的变化

这次升级有三个东西是纸面数据看不出来的。

第一个是新的 xhigh 推理等级。之前 Claude Code 里只有 low、medium、high 三档，现在多了个 xhigh。实测下来，对复杂架构级任务确实有帮助，模型会花更多 token 去"想"，而不是急着给你一个看起来对的答案。

第二个是自适应思考 (Adaptive Thinking) 成了默认模式。这东西的逻辑是让模型自己决定在哪些地方深入推理、哪些地方快速回答。听起来很聪明对吧？

问题就出在这。

第三个是新的分词器。输入 token 数会增加 1.0 到 1.35 倍，虽然 Anthropic 说推理效率提升了，整体 token 消耗反而降了最多 50%，但你的账单上会看到一些"微妙的变化"。坦率讲，这种"价格没变但用量变了"的操作，需要自己跑几天再下结论。

## HN 评论区里的真实声音

我花了不少时间泡在那 1421 条评论里，挑几个最有信息量的。

用户 JamesSwift 反馈说升级后模型表现明显退步，折腾了一番发现，关掉自适应思考、手动把 effort 拉到最高，性能才恢复到 4.6 水平。Anthropic 员工 Boris 在评论区回复承认"自适应思考目前确实还不完善"。

这就很有意思了。一个还没完善的功能，被设成了默认开启，而且没有关闭开关。

另一边，用户 arcanemachiner 说在 275K token 的超长上下文里，4.7 表现"出乎意料地好"，比 4.6 在 200K 左右就开始犯糊涂要强很多。Jeremy Howard，就是 fast.ai 的那位，说这是"第一个真正理解我意图的模型，它愿意跟我讨论而不是直接碾过去"。

还有个很有意思的测试。有人用"洗车问题"来考模型，问你家离洗车店 50 米，该走路去还是开车去。正确答案当然是开车，因为你要洗的是车。4.7 有时候能答对，有时候答错然后编一套理由出来。这种不一致性暴露了自适应思考在"非典型推理"上的短板。

Cursor 团队倒是给了正面反馈，内部编码基准从 58% 涨到 70%。Notion 也说工具调用错误减少了三分之一。

## 隐藏在讨论中的几个信号

HN 用户 ljm 指出了一个很少有人注意的问题，Anthropic 的产品线正在碎片化。Chat、Cowork、Code 三个入口各自为政，项目不互通，配额消耗逻辑不一样，MCP 在 Chat 里能用但 Code 里不行。这种体验割裂在快速迭代中越来越明显。

还有个争议点，思考过程 (thinking traces) 现在默认不显示了，要手动加参数才能看到。用户 MarkMarine 认为这是 Anthropic 为了防止模型蒸馏而牺牲透明度，跟之前强调的安全承诺有矛盾。

用户 lambda 做了个有趣的对比，用本地跑的 Qwen 3.5 122B 和 Gemma 4 31B 去做同样的逻辑题，这两个小模型反而稳定答对了 Opus 4.7 答错的题目。

这说明什么？模型越大不代表每个维度都越强。自适应思考在节省算力和深度推理之间找平衡，有时候会在"简单但非典型"的问题上翻车。

## 我的判断

我认为 Opus 4.7 是一次"实质性但不均匀"的升级。

编码能力确实涨了，而且是体感能感知到的那种涨。长上下文处理、视觉理解、文档分析，这些方向进步明显。如果你主要用 Claude Code 写代码，升级是值得的。

但自适应思考这个功能，我的判断是它还不该被设成默认开启。它在标准编码任务上表现不错，但在需要"慢想"的场景下会偷懒。目前的解决方案就是手动 `/effort xhigh`，很土，但有效。

说实话我也不确定 Anthropic 为什么急着把一个自己员工都承认"还没完善"的功能推成默认。可能是产品节奏的压力，可能是数据显示大多数场景确实更好了。但对于重度用户来说，这种"平均更好但极端场景变差"的 tradeoff 是需要自己去摸索的。

文档理解的提升倒是让我意外。图表识别从 13.5% 到 55.8%，这个幅度意味着之前基本不能用、现在勉强能用了。不过 LlamaIndex 的 Jerry Liu 提醒说，企业文档管线用 Opus 做 OCR 级别的工作，每页大约 7 美分，成本上还是不如专用方案。

## 你该怎么做

如果你是 Claude Code 用户，现在就可以试。在 `.claude/settings.json` 里确认你用的是最新模型，然后跑你最熟悉的项目感受一下差异。遇到模型"变懒"的情况，试试 `/effort xhigh`。

如果你在生产环境用 API，别急着切。新分词器会改变 token 计数，先拿你的实际 prompt 测一轮成本变化。

如果你在做文档处理相关的应用，值得重新评估一下 Claude 在你管线里的位置，视觉能力的提升可能让一些之前不可行的方案变得可行了。

1934 个赞说明社区对 Anthropic 的期待还在。但期待和满意之间，隔着一个自适应思考的开关。
