Karpathy 前两天发了条推，说自己差点被供应链攻击搞了。对，就是那个前 Tesla AI 总监、OpenAI 创始成员的 Karpathy。连他都差点中招，你我呢？

这次的目标是 axios，npm 上每周下载量超过 3 亿次的 HTTP 客户端库。3 亿。你手头的项目大概率直接或间接依赖了它。

## 到底发生了什么

Karpathy 说他前几天在折腾 Gmail 和 Google Calendar 的命令行工具，装了一个叫 `googleworkspace/cli` 的包。他后来扫描系统，发现这个包解析到的版本是 1.13.5，恰好没被感染。

但问题在于，这个项目的依赖没有锁定版本号。

也就是说，如果他晚几个小时装，npm 会自动拉取最新版本，而那个最新版本已经被投毒了。"I'd be pwned"，他原话。

这就是供应链攻击最阴险的地方，你什么都没做错，只是装了个包，在一个错误的时间。

## 为什么 npm 和 pip 的默认行为是个隐患

坦率讲，这个问题的根源不在 axios，也不在那个被投毒的包，在包管理器本身的设计哲学。

npm install 的默认行为是什么？拉最新版本。pip install 呢？一样，拉最新版本。没有版本锁定，没有发布时间窗口校验，没有任何"这个版本刚发布 2 小时你确定要装吗"的提醒。

Karpathy 自己也说了，个人层面你可以做一些防护，比如设置 release-age 约束（只安装发布超过一定天数的版本），或者用容器隔离。但他认为这些都是补丁，真正需要改变的是包管理器的默认行为。

我自己也还在摸索最佳实践，但有一个体感很明确，npm 和 pip 的"默认信任一切"模式，在 2026 年已经不合理了。十年前开源生态还没这么大，供应链攻击也没这么频繁，默认拉最新版本是效率优先的合理选择。但现在？攻击者就是利用这个"发布到被发现"的时间窗口来大规模感染用户。

## 社区里的多种声音

这条推下面的讨论很有意思。

有人提出应该强制所有 npm 包启用二因素认证，这样攻击者至少没法轻易劫持维护者账号来推送恶意版本。也有人反驳说，很多投毒根本不是账号被劫持，而是通过 typosquatting（注册一个名字和热门包很像的包）或者依赖混淆来实现的，二因素认证防不了这种。

还有一派观点认为，应该在包管理器层面加一个"冷却期"，新发布的版本默认不被解析为 latest，等 24-48 小时安全扫描通过后再上线。这个思路我觉得挺靠谱，但实际落地有个矛盾，紧急安全补丁本身就需要快速发布和安装。

说真的，没有银弹。

## 我的判断

我认为 Karpathy 说到了点子上，但他的建议还不够激进。

"改变包管理器的默认行为"听起来合理，但 npm 和 pip 背后的组织（npm Inc 被 GitHub 收购后归微软，pip 归 PyPA）推动这种根本性改变的动力一直不足。因为改默认行为意味着破坏向后兼容性，意味着无数 CI/CD 流水线要改，意味着社区会骂娘。

反正我觉得，短期内指望包管理器替你兜底是不现实的。

真正应该变成行业标配的做法是，每个项目都必须有 lockfile，并且 CI 环境下只用 `npm ci` 而不是 `npm install`。lockfile 的意义就在于，它把"某一刻经过验证的依赖树"固化下来，后续安装严格按这个树来，不会因为上游发布了新版本就自动拉取。

你想想看，Docker 镜像我们都知道要固定 tag，甚至固定 digest。但 npm 包呢？多少人的 package.json 里写的是 `^1.13.0` 这种允许自动升级的版本范围？

这才是真正的风险敞口。

## 你现在就能做的几件事

**锁版本。** 检查你的 package.json，把 `^` 和 `~` 去掉，或者至少确保 package-lock.json 被提交到了 git 仓库里，并且 CI 用 `npm ci`。

**设置 release-age 约束。** npm 可以通过 `.npmrc` 配置，pip 可以用 `pip-audit` 这类工具在安装前扫描。

**定期跑依赖审计。** `npm audit` 和 `pip-audit` 都是免费的，加到 CI 里就是一行配置的事。

**关注 Socket.dev。** Karpathy 推文里链接的那篇详细分析就来自 Socket，他们专门做供应链安全检测，对 npm 和 PyPI 的覆盖比较全。

Karpathy 差点被搞了，但他运气好，装的版本恰好在投毒之前。下一次你装包的时候，运气会不会也这么好？
