# 96.6% 召回率的 AI 记忆系统，我翻了翻代码发现事情没那么简单

上周一个叫 MemPalace 的开源项目在 GitHub 上两天冲到 23000 星。它的 README 写得很猛，"史上 benchmark 得分最高的 AI 记忆系统，完全免费，零 API 调用"。

我当时的反应是，又来？

但点进去看了架构图，我愣了几秒。这玩意儿的思路确实和市面上的 Agent 记忆方案不太一样。

## 先说清楚它在解决什么问题

你用 Claude Code 写了半年代码，换一个 session，AI 完全不记得之前发生了什么。你做的技术决策、踩过的坑、选型的理由，全没了。

现在主流的解决方案大概两条路。一条是让 LLM 自己做摘要，把对话压缩成"记忆"存起来。Mem0、Zep 走的都是这条路。问题是摘要会丢信息，LLM 觉得不重要的细节可能恰恰是你后面需要的。

另一条是 RAG，把对话原文切块做向量检索。但对话一多，检索质量直线下降。

MemPalace 选了第三条路，原文全存，但用一套空间隐喻来组织。

## "记忆宫殿"到底是怎么搭的

MemPalace 借用了古希腊的"记忆宫殿"记忆术，把记忆按空间结构分层。

最外层叫 Wing（翼），对应一个项目或一个人。Wing 下面是 Room（房间），比如 auth、billing、deployment。Room 里面有 Hall（厅），按记忆类型分成 facts、events、discoveries 几类。最底层是 Drawer（抽屉），存的就是原始对话原文。

还有个叫 Tunnel（隧道）的东西，用来连接不同 Wing 里相关的 Room。

技术上，底层是 ChromaDB 做向量嵌入加语义检索，SQLite 存知识图谱和时间线事实。全部本地运行，不依赖云服务。

这套分层组织有什么用？数据很直观。全局搜索的 R@10 是 60.9%，限定到某个 Wing 内搜索涨到 73.1%，再加上 Room 过滤直接到 94.8%。相比平铺检索，提升了 34 个百分点。

坦率讲，这个思路让我眼前一亮。不是靠更好的 embedding 模型，也不是靠更复杂的 reranking，而是靠元数据过滤缩小搜索范围。简单粗暴，但有效。

## 那个 96.6% 的 benchmark 到底可不可信

这才是最有意思的部分。

MemPalace 在 LongMemEval 这个专门测 AI 长期记忆的 benchmark 上拿到了 96.6% 的 R@5，号称历史最高。README 甚至还提了一个 100% 的 hybrid 模式得分。

我认为这个数字本身大概率是真的，有人在 M2 Ultra 上独立复现了，5 分钟跑完。

但这里有几个坑。

第一，96.6% 是"raw 模式"的成绩，也就是存了完整原文去检索。这当然分高，你把答案原封不动存着，搜到的概率肯定大。但实际用的时候，谁能承受把所有对话原文塞进 context window？

第二，MemPalace 还有个叫 AAAK 的实验性压缩方案。一旦开了压缩，分数直接掉到 84.2%，降了 12 个百分点。README 里对这一点的处理很微妙，96.6% 放在最显眼的位置，AAAK 的回退藏在下面。

第三，有开发者在 HN 上指出，那个 100% 的 hybrid 模式用了 top-k=50 加 reranking，测试集本身 session 数都没 50 个。这就像考试范围只有 10 道题，你把 50 道都背了，说自己考了满分。

说真的，这种 benchmark 营销在开源圈越来越常见了。项目可能确实好用，但分数包装得太 aggressive，反而伤害信任。

## 社区里的多种声音

HN 上的讨论挺分裂的。

有人在实际测试后给了正面评价，说它"一行命令就能把对话历史导入 ChromaDB，完全本地，MIT 协议"，准备集成到自己的工作流。

也有人提出了根本性的质疑。一个叫 Lodestar 的竞品开发者说了一句很到位的话，MemPalace 优化的是"检索质量"，但 Agent 记忆真正该优化的是"决策辅助质量"。存了原文能搜到是一回事，搜到的东西能不能帮 Agent 做出更好的决策是另一回事。

还有人提了个很现实的问题，对话越积越多，全量存储的检索效率怎么保证？今天你有 100 个 session，检索很快。明年有 10000 个呢？

另外这个项目还有一些工程上的硬伤，比如 fact_checker.py 这个做矛盾检测的模块写了但没接入知识图谱，macOS ARM64 上有 segfault 的 issue，hooks 里还被人报了 shell 注入漏洞。

## 我的判断

我认为 MemPalace 的核心直觉是对的，用结构化的空间组织替代纯向量检索，这个方向值得探索。34% 的检索提升不是小数字。

但我也认为，它现在更像是一个概念验证，而不是一个生产就绪的系统。

反正我觉得，Agent 记忆这个赛道最大的问题根本不是"怎么存"和"怎么搜"，而是"存什么"。当你把所有对话原文一股脑塞进去，本质上是把筛选的成本从写入时转移到了读取时。短期内数据量小的时候体验很好，长期一定会撞墙。

怎么说呢，可能最理想的方案是把 MemPalace 的空间组织思路和传统摘要方案结合起来。用结构化的 Wing/Room 做路由，但每个 Room 里同时保留原文和 LLM 生成的结构化摘要，检索时先搜摘要再回溯原文。

如果你现在就需要一个 Agent 记忆方案，MemPalace 值得跑一下看看。安装很简单，`pip install mempalace` 然后 `mempalace init` 就行。它支持 MCP 协议，Claude Code、Cursor、ChatGPT 都能接。一年成本大概 10 美元，对比 LLM 摘要方案的 507 美元/年，确实便宜得多。

但别被 96.6% 那个数字催眠了。跑一下你自己的对话数据，看看在你的使用场景下检索质量到底怎么样，这比任何 benchmark 都靠谱。

记住开头我说的那个反应吗？"又来？"

我现在的想法变了，但只变了一半。这不是又一个刷分项目，架构思路确实有新东西。但它也远没到"史上最强"的程度。开源社区最不缺的就是 bold claim，最缺的是 honest claim。
