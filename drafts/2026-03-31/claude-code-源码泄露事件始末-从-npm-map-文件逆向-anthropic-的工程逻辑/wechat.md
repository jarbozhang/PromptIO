# 从泄露的 Claude Code 源码，逆向拆解 Anthropic 的 Agent 架构与工程底线

最近几天，AI 编程赛道的竞争已经到了白热化阶段，但 Anthropic 内部究竟是如何构建 Claude Code 这样一个复杂的终端 Agent 的？一直是个黑盒。前几天，有人通过 NPM 包里的 source map 文件把 Claude Code 的底层逻辑扒了个底朝天。抛开吃瓜和猎奇，这可能是我们第一次能如此近距离、成体系地拆解一家顶级 AI 公司的内部 Prompt 工程与 Agent 架构设计。对于正在卷 AI Coding 赛道的国内开发者和创业者来说，这是一份价值连城的“对齐”资料。

这次泄露的源头并不高深。JavaScript/TypeScript 项目在构建打包时，如果配置不当或出于调试目的保留了 `.map` 文件， anyone 都可以通过浏览器或简单脚本将压缩混淆后的代码还原。Claude Code 本质上是一个通过 NPM 安装的 Node.js 命令行工具，于是被人在包里翻出了完整的未混淆源码。这不仅是个安全插曲，更给全行业上了一堂生动的工程课。

在这套泄露的代码里，最让我受启发的不是他们用了什么惊天动地的新算法，而是 Anthropic 极度克制且务实的系统设计。我们先看核心的架构逻辑：Claude Code 采取了非常典型的“大一统底层 + 强力路由”设计。它并没有像很多开源项目那样，为不同语言或任务注册海量的微调工具，而是极其依赖底层模型（Claude 3.5 Sonnet）强大的指令跟随与泛化能力。

模型能够通过 Bash 执行 Shell 命令、通过文件读写工具直接在本地操作，这种方式把底层的系统能力直接暴露给了大模型。这意味着 Claude Code 在架构上认定了一个前提：只要基座模型足够聪明，你不需要包一层复杂的抽象，模型自己就是最好的任务编排器。

而在 Prompt（系统提示词）方面，Anthropic 做了大量极其细腻的“护栏”工程。比如，在处理 Git 操作时，代码里有非常明确的防范提示词，严防模型执行 `git push --force` 或者直接 drop 掉用户的本地提交。它会强制模型在进行高风险文件操作前生成 diff，并要求用户确认。这套逻辑完美契合了 Anthropic 一直标榜的“安全优先”基因，而且这种安全不是泛泛而谈的道德说教，而是深入到了每个具体系统调用中的硬性规则。

在处理大型代码库的上下文时，他们构建了一套很实用的文件层级管理机制。系统会指引模型优先通过目录结构和特定后缀文件（如 `README.md` 或 `.cursorrules`）来建立全局认知，然后再深入具体文件。这非常像人类工程师接手一个新项目的做法：先看架构，再抠细节。虽然业界都在吹捧复杂的 RAG 架构，但在端侧 Agent 场景下，Claude Code 明显更倾向于让大模型依靠文件系统的全局索引自主探索。

在英文社区里， Hacker News 和 X（推特）上的讨论已经远远超越了“看热闹”的层面，而国内中文信息圈目前似乎还停留在“Anthropic 出现安全事故”的新闻速报阶段。HN 上最火的一个洞察是：**高级 AI Agent 系统的核心护城河，其实早就不是那些巧妙的算法逻辑了，而是枯燥的工程实现和那几万行不断迭代的 System Prompt。**

很多国外开发者意识到，把这套系统提示词和工具调用逻辑拿去套在 GPT-4o 甚至开源的 Llama 3 上，只要稍微做些对齐微调，也能跑出一个八九不离十的 Coding Agent。这印证了一个残酷的行业趋势：软件层面的 AI Agent 护城河极浅，真正的壁垒依然在于模型底层能力的拉扯，以及海量真实开发者交互数据带来的系统飞轮。

更深层次的讨论集中在安全边界上。黑客们发现，Claude Code 的架构极度依赖单次对话上下文进行决策。如果缺乏长期记忆中更高级别的全局权限验证，理论上极易被“内部投毒”。比如，你在代码仓库里埋入一段伪装成普通配置文件的恶意指令，要求 Agent 执行某个隐蔽的 bash 脚本。面对越写越长的上下文窗口，模型很容易在不知不觉中遗忘初始化时的安全边界，顺从恶意指令。这种 Agent 级别的越狱，是所有做端侧 Agent 的团队接下来必须面对的深坑。

我认为，这次泄露对国内正在卷 AI Coding 工具（比如各种 Copilot、Cursor 的国产平替）的团队来说，是一记警钟，也是一张明牌。Anthropic 的工程逻辑证明了：**做 AI 编程工具，大包大揽的中间层往往是过度设计。** 你的重点不该是写一套无比复杂的逻辑去猜测程序员的意图，而是要让模型学会像程序员一样使用底层的终端工具。

同时，这也打破了很多人对“系统提示词”的迷信。Anthropic 写的提示词确实极其精巧，但在系统的整体表现中，这只占了不到三成的功劳。剩下的七成完全依赖于 Claude 3.5 Sonnet 本身在长上下文理解、指令遵循上的强悍基座能力。如果基座模型不够聪明，哪怕你把 Anthropic 的源码原封不动抄过来，跑出来的体验也会是一坨散沙。

对于国内的基座模型厂商来说，Claude Code 的这套源码（现在 GitHub 上已经有人做了完整的带注释还原项目）绝对是不可多得的评测与对齐标准。我强烈建议模型团队直接把这套系统提示词和工具调用逻辑作为内部测试集，看看自家的模型在严格限制危险操作的前提下，到底能走通多少个多步骤的代码重构任务。

最后，无论你是做前端的还是做 AI 应用的，赶紧去查查自己公司的生产环境打包脚本。确保在 release 模式下彻底关闭 Source Map 生成。连 Anthropic 这种顶级安全水平的公司都会栽在构建配置的细节上，千万别觉得这种事轮不到你。
