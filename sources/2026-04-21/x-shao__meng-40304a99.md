MCP 会被 Skills 取代？MCP 会被 CLI 取代？未必！

Anthropic 技术成员、MCP 联合创始人 @dsp_ 在 @aiDotEngineer Europe 主题演讲「The Future of MCP」，关于今年 AI Agent 的未来发展，以及 MCP 在其中的核心定位：2025 年是 Agent 的探索与验证期，2026 年将转向生产落地期，MCP 正是支撑这一转变的“胶水层”。

MCP 的核心价值与愿景
MCP 为 AI Agent 设计的可扩展连接层。它允许开发者构建 MCP 服务器，使 Agent（或人类用户）能动态发现并使用工具，同时支持结构化输出、UI 呈现和语义丰富的描述。

MCP 的成功在于其“无聊但实用”的规范特性：它像一个通用端口，让任何开发者都能为特定领域或应用扩展 Agent 能力，而无需从零构建专有集成。目前，MCP 生态已展现强劲增长，包括主流框架的依赖集成，以及远程授权、任务原语等功能的扩展。

2026 年 Agent 的连接性堆栈
David 提出，构建 2026 年生产级 Agent 的最佳实践是采用混合连接栈，而非依赖单一技术：

· Skills：提供领域特定的知识包、提示词和可复用能力，适合上下文密集但结构相对固定的场景。
· MCP：作为富语义层，提供结构化工具、动态 UI 和双向交互，适用于需要深度集成、审计和复杂编排的企业或跨系统任务。
· CLI / Computer Use：处理本地命令执行、GUI 自动化等简单或底层操作。

最佳 Agent 会根据任务需求无缝组合三者。Skills 处理领域知识，MCP 负责企业数据连接，CLI/Computer Use 补充本地执行。

解决核心工程痛点：上下文膨胀与工具发现
Agent 面临的最大实际挑战之一是上下文窗口限制。传统方式下，加载数十个工具的完整描述可能消耗大量 token，甚至占上下文的显著比例。
David 重点介绍了渐进式发现作为解决方案：

· Agent 不再一次性加载所有工具描述，而是按需发现和加载。
· 通过工具搜索机制或延迟披露，仅引入当前任务相关的工具，大幅降低上下文占用（示例中显示显著优化）。
· 这与 Skills 的渐进披露机制类似，形成互补。

此外，演讲强调程序化工具调用与 Agent 编排：Agent 应能自主组合工具、生成并执行代码（类似 REPL），并利用 MCP 的结构化输出减少无效调用次数，提升可靠性和效率。

Agent 与 MCP 服务器的设计最佳实践
· 对于 Agent 开发者：避免简单地将现有 REST API 1:1 映射到 MCP。应从 Agent 的使用视角重新设计接口，提供语义丰富、易于编排的工具描述。
· 对于服务器作者：充分利用 MCP 的丰富语义，不仅暴露工具，还应主动提供 MCP 应用（动态 UI）和打包的 Skills，让能力更易被 Agent 消费。

这一转变体现了思维模式的变化：从“暴露 API”转向“为 Agent 设计体验”。

MCP 协议的未来路线图与企业就绪
演讲展望了 MCP 的核心改进方向：
· 传输层与扩展性：优化当前 streamable HTTP，支持无状态传输等新协议，解决大规模部署下的性能瓶颈。
· SDK 演进：推出 TypeScript 和 Python 的 SDK v2，提升开发者体验。
· 企业特性：包括跨应用访问、Server Discovery（Agent访问网站时自动发现关联的 MCP 服务器，实现“即访即用”），以及增强的授权和审计机制。
· 扩展机制：Skills over MCP ——服务器直接打包并持续更新领域知识、提示词和复合能力，无需依赖中央注册表，实现去中心化且始终最新的能力交付。

这些路线图直面生产落地痛点：安全性、可扩展性、上下文管理与企业集成。David 特别提到，2026 年的重点是让 MCP 从“可用”走向“企业级可靠”，包括异步任务增强和触发器等功能。
