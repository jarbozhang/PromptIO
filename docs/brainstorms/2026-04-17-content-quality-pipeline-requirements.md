# 内容质量管线升级：从"生成即完成"到"生成-批评-精修"闭环

## Problem Frame

PromptIO 管线的采集端（RSS/GitHub/arXiv/X）和编辑标准（wechat.md L1-L5 五层质检）都已经很成熟，但中间存在一个断层：**文章生成后直接 commit，没有任何自动质检或迭代**。wechat.md 定义了 L1-L5 五层检查标准，gates.js 在测试中被引用但实际不存在。

饼干哥哥的文章验证了一个核心观点：自我批评循环让输出质量提升 10 倍，"让 AI 删不让 AI 加"比"让 AI 补充"有效十倍。

当前管线等效于饼干说的"1.0 水平"（单次生成），尽管 prompt 质量已经是 3.0 级别。需要补齐的是**执行层的闭环**。

**核心假设风险**：饼干文章的"自我批评循环提升 10 倍"是经验性声称，未在本管线验证。LLM 自我批评对主观维度（如"活人感"）的效果存疑。**实施前必须做 pilot 验证**（见 Pre-flight 步骤）。

**同模型偏见风险**：QA agent 和生成 agent 使用同一个 Claude 模型，即使隔离上下文，仍共享相同的系统性偏见。QA agent 可能系统性认可同模型生成的 AI 风格。需要用实际发布数据校准 QA 评分的有效性。

## 当前工作流 vs 目标工作流

```
当前：采集 → 评分选题 → 10个子代理并行生成 → commit
                                    ↑ 一步到位，无迭代

目标：采集 → 评分选题 → 生成初稿 → 独立质检 → 质修循环(≤3轮) → [人工审核] → commit
                                    ↑ 多步迭代，质量闭环        ↑ status: approved 才 commit
```

## Pre-flight — 实施前必须完成的验证

- **PF1. Pilot 验证质修循环有效性**：取 5 篇现有 drafts 中质量偏低的文章，手动运行一轮 QA → 修改循环，测量 L4 评分是否实际提升，以及修改后是否能对应到人工审核通过。如果 5 篇中 < 3 篇有可感知改善，推迟整个升级，先优化生成 prompt
- **PF2. QA 评分校准测试**：取 5 篇已知高传播（实际阅读量 top）文章和 5 篇低传播文章，让 QA agent 分别评分。如果 QA 评分无法区分高低传播文章（两组均分差 < 1 分），则 L4 >= 7 的阈值是虚假安全感，需要重新设计评分标准
- **PF3. 记录当前管线时间基线**：跑一次完整的 10 篇生成管线，记录从开始到全部 commit 的总耗时，作为"2x"约束的基线
- **PF4. gates.test.js 处理**：在 P0 工作开始前，删除 test/gates.test.js 或为其添加 skip wrapper，否则 npm test 会因 gates.js 不存在而崩溃

## Requirements

### P0 — 核心闭环（必须实现）

**质检自动化**

- R1. 每篇文章生成后，**先做 L1 机械替换**（禁用词→替代词、禁用标点→逗号/句号，按 wechat.md L1-1/L1-2 规则直接替换，不消耗修改轮次），然后由**独立的质检 agent**执行 L1 结构性套话和空泛工具名检查。输出违规项列表和修复建议
- R2. 独立质检 agent 执行 L2 风格一致性检查：开头是否从具体事件切入、是否有长短句交替和单句成段、是否使用了口语词组、是否有自嘲或承认不足
- R3. 独立质检 agent 执行 L3 内容质量检查：每个核心观点是否有具体支撑、知识输出方式、对立面理解、原型专项检查
- R4. 独立质检 agent 执行 L4 活人感终审：整体是否像"工程师聊天"而非"AI 输出信息"。输出**二元判断（pass/fail）+ 具体"AI 味段落"位置列表和修改建议**。不使用连续数字评分（LLM 在 6.8 vs 7.2 级别的 test-retest reliability 低，不适合做触发阈值）
- R5. 独立质检 agent 执行 L5 传播潜力检查：标题品牌/利益点/可操作性

**自我批评循环**

- R6. 质检结果低于阈值（L1 结构性套话/空泛工具名任一违规，L2-L3/L5 任一项 < 7/10，L4 判定 fail）的文章，自动进入"质修循环"，最多 3 轮。每轮：质检 agent 出具问题清单 → 写作 agent 执行修改 → 质检 agent 验证。**质检和修改必须由不同的 agent 完成**，避免自评自改。**3 轮后仍未达阈值的草稿，带上完整质检历史进入人工审核 gate，不再自动循环**。**错误处理**：QA agent 返回不可解析输出或超时，视为该轮失败，消耗 1 轮配额；3 轮均错误则连同错误日志进入人工审核
- R7. 质修循环中的修改原则：
  - **删减类**（冗余/节奏/风格）：只删不加，禁止扩充
  - **替换类**（质检指出的结构性缺陷，如缺少个人故事、缺少对立面理解）：允许定向替换，条件为 (1) 质检 agent 明确指出具体段落和缺陷类型，(2) 替换内容与原段落功能相同，(3) 替换后全文总字数 ≤ 原文总字数 × 1.1（允许 10% 弹性，以容纳口语化碎片和节奏断裂所需的额外空间）
  - 违反上述条件的修改视为"加内容"，禁止

### P1 — 合并到质检中（R1/L4 的增强项）

**去 AI 味专项检测**

- R10. 在 L1 扫描中增加去 AI 味 6 项检查：三段式结构重复、连续数字列表超 2 个、泛泛总结式结尾、缺少"我"的视角、所有观点平衡中立、"删掉最后一段是否更好"测试
- R11. 检测到 AI 味模式时，将违规段落位置和类型并入 R6 质修循环的问题清单（不单独触发重写，消耗 R6 的轮次配额），由写作 agent 按 R7 替换规则执行修改

### P2 — 延后实施（需 REACH 数据验证开头是瓶颈后再做）

**开头/结尾多版本竞争**

- R8. 初稿生成后，为每篇文章额外生成 5 个开头候选（覆盖好奇/恐惧/惊喜/共鸣/反常识 5 种情绪触发）和 3 个结尾候选（首尾呼应/个人表态/数据收尾）
- R9. 用独立的评估 agent 对开头和结尾候选评分，自动替换为最高分版本

### P3 — 延后至 REACH >= 9 样本量达到 ~20 篇

**历史爆款反向提炼**

- R12. 每周自动分析 article-registry.md 中 REACH >= 9 的文章，提取共同特征（标题模式、开头类型、结构特征、人格化浓度、话题类型），输出到 wiki 中作为选题和写作参考
- R13. 将提炼出的爆款规律反哺到 scoring.md 和 wechat.md 的权重调整建议中

## Success Criteria

- 每篇文章在进入人工审核 gate 前自动通过 L1-L5 质检，质检报告写入 meta.yaml
- L4 活人感 pass 率 >= 70%（质检 agent 二元判断，非自评。用 PF2 校准测试验证评分有效性后启用）
- 人工审核通过率提升（PF3 建立时间基线后，第一周记录人工审核通过/打回比例作为质量基线）
- 生成到 commit 的管线耗时增加不超过 2 倍（基于 PF3 记录的基线。假设 >= 60% 文章首轮通过质检；如果 pilot 显示触发循环比例 > 50%，需要调低阈值或并行化 QA）
- **外部验证**：发布后追踪实际微信阅读量/点赞数，建立 REACH 自评分与实际表现的相关性数据（手动记录即可，不需要 API 对接）

## Scope Boundaries

- 不改变现有的采集管线（pipeline.js fetch-only 逻辑不动）
- 不改变现有的评分/选题流程（仍由 Claude Code 会话完成）
- 不新增外部依赖（质检和精修全部通过 Claude Code 子代理完成）
- 不自动发布，仍然保持人工审核 gate
- 不涉及公众号后台 API 对接

## Key Decisions

- **质检在 Claude Code 会话内完成，不需要独立脚本**：当前生成已经在 Claude Code 会话中通过子代理完成，质检循环作为生成流程的延伸，在同一会话中完成。不需要写 gates.js，直接用 prompt 驱动。**现有 test/gates.test.js 中引用的 gates.js 接口（titleScorer/goldQuote/summaryGen）暂不实现，规划阶段决定是删除测试还是后续补实现**
- **质检 agent 必须独立于生成 agent**：避免"自评自改"。质检 agent 仅接收最终初稿 markdown 作为输入（不含生成 prompt 或上下文），基于 wechat.md 标准以独立视角重新审视。质检结果以问题清单形式传给修改 agent，修改 agent 只看到问题清单和原文，不看到质检推理过程
- **精修原则：正文"只删不加"，结构缺陷允许"定向替换"**：纯冗余/节奏问题只做减法；结构性问题允许替换段落，全文总字数 ≤ 原文 × 1.1（10% 弹性容纳口语化改造）
- **L1 分两层执行**：L1-1 禁用词和 L1-2 禁用标点由机械替换直接处理（不消耗修改轮次）；L1-3 结构性套话和 L1-4 空泛工具名由质检 agent 检查
- **L4 使用二元判断而非连续评分**：pass/fail + 具体段落位置列表。避免 LLM 在评分边界的不稳定性导致循环触发随机化
- **开头/结尾竞争在初稿之后进行**：不在初稿生成时就做多版本，而是先有完整初稿，再单独优化薄弱环节
- **质检阈值需要迭代确定**：初期建议 L1 结构性套话/空泛工具名零容忍（机械替换已处理禁用词/标点）、L2-L3/L5 各项 >= 7/10、L4 pass/fail。3 轮后仍未达标则进入人工审核。运行一周后根据实际触发循环比例和人工审核反馈调整阈值。如果首轮通过率 < 60%，优先降低阈值而非增加循环轮次

## Dependencies / Assumptions

- 假设现有 wechat.md 的 L1-L5 标准已经足够全面。质检 agent prompt 从 wechat.md 提取标准，以独立视角重新评估。**但 PF2 校准测试可能证明同模型评分无效，届时需要引入外部校准锚点或人工评分样本**
- 假设 10 个子代理并行生成的架构不变，质检循环在每个子代理流程末尾串行执行（10 篇文章的质检并行进行，单篇内的质修循环串行）
- 依赖 article-registry.md 持续维护准确的 REACH 评分数据
- REACH 自评分仅作为内部参考，不作为唯一成功指标；需要人工审核通过率和实际发布数据作为外部校准
